01 — Contexte & objectifs

Ce projet personnel consiste à monter un Security Operations Center (SOC) en miniature sur VirtualBox. L'objectif est de simuler des attaques réelles (brute force, scan réseau, mouvement latéral) depuis une machine Kali Linux, de collecter les événements Windows dans Splunk via des Universal Forwarders, puis de créer des alertes et dashboards pour détecter ces attaques en temps réel — exactement comme le ferait un analyste SOC en entreprise.

Ce projet est directement aligné avec le parcours TryHackMe SOC Level 1 et les compétences recherchées pour un rôle d'analyste SOC ou d'ingénieur cybersécurité.

03 — Installation & configuration Splunk
1
Installer Splunk Enterprise sur Ubuntu
Téléchargement du package .deb Splunk Enterprise Free (500 Mo/jour gratuit). Démarrage du service et accès à l'interface web sur http://192.168.2.10:8000
2
Installer l'Universal Forwarder sur Windows Server
Le Splunk Universal Forwarder collecte et transmet les Windows Event Logs (Security, System, Application) vers le serveur Splunk en temps réel via le port 9997.
3
Configurer les inputs — Event Logs Windows
On configure le forwarder pour surveiller le canal Security (EventID 4624, 4625, 4648, 4688...), System et Application. Toutes les connexions, échecs et processus sont transmis à Splunk.
4
Vérifier l'indexation des données
Dans Splunk : Search & Reporting → requête index=windows_events → les logs doivent apparaître en temps réel.
Installation Splunk & Universal Forwarder
# Sur VM-Splunk (Ubuntu Server)
wget https://download.splunk.com/products/splunk/releases/9.x.x/linux/splunk-9.x.x-linux-2.6-amd64.deb
sudo dpkg -i splunk-9.x.x-linux-2.6-amd64.deb
sudo /opt/splunk/bin/splunk start --accept-license
sudo /opt/splunk/bin/splunk enable boot-start

# Sur VM-Windows — Universal Forwarder
# inputs.conf — Collecte des logs Windows Security
[WinEventLog://Security]
index = windows_events
sourcetype = WinEventLog:Security
disabled = false
renderXml = true

# outputs.conf — Envoi vers Splunk
[tcpout:splunk_indexer]
server = 192.168.2.10:9997

04 — Simulation des attaques

Les attaques sont lancées depuis VM-Kali pour générer des événements de sécurité détectables dans Splunk.

A1
Brute Force SSH / RDP avec Hydra
Hydra lance des centaines de tentatives de connexion avec une wordlist (rockyou.txt). Chaque échec génère un EventID 4625 (échec logon) sur Windows, visible dans Splunk.
A2
Scan réseau avec Nmap
Scan de ports agressif depuis Kali (-sS, -sV, -O). Génère du trafic réseau anormal détectable par Splunk + Windows Firewall logs.
A3
Tentative de mouvement latéral
Simulation de connexion avec un compte compromis depuis Kali vers Windows (EventID 4648 — logon avec identifiants explicites). Typique d'un attaquant qui se déplace dans le réseau.
kali@lab — Simulation attaques
# A1 — Brute Force RDP avec Hydra
hydra -l Administrator -P /usr/share/wordlists/rockyou.txt \
      rdp://192.168.2.20 -t 4 -vV

# A2 — Scan réseau agressif
nmap -sS -sV -O -A --script=vuln 192.168.2.20

# A3 — Connexion avec identifiants explicites (mouvement latéral)
evil-winrm -i 192.168.2.20 -u admin -p Password123

# Vérification dans Splunk :
# index=windows_events EventCode=4625 | stats count by src_ip

05 — Détection Splunk — Requêtes & Alertes

Requêtes SPL (Splunk Processing Language) créées pour détecter chaque type d'attaque :

Détection brute force — EventID 4625 (> 50 échecs en 5 min)
index=windows_events EventCode=4625 | bucket _time span=5m | stats count by _time, IpAddress, TargetUserName | where count > 50 | sort -count
Détection scan réseau — connexions multiples ports distincts
index=windows_events EventCode=5156 | stats dc(DestPort) as ports_scanned by SourceAddress | where ports_scanned > 20 | sort -ports_scanned
Mouvement latéral — EventID 4648 (logon avec identifiants explicites)
index=windows_events EventCode=4648 | stats count by SubjectUserName, TargetServerName, IpAddress | where SubjectUserName != TargetServerName | sort -count
