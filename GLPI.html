Mission 1 — Le réseau inventorié
Mission 1Inventaire du parc
Mise en place d'une solution d'inventaire automatisée représentative des systèmes TiersLieux86 et AuditME. La solution remonte les inventaires complets (matériel, OS, logiciels, IP/MAC, utilisateurs) et produit un fichier CSV.

Contexte & objectifs de l'inventaire
Périmètre : 4 machines — 2 clients + 2 serveurs

L'inventaire doit couvrir l'ensemble du parc TiersLieux86 et AuditME. La solution retenue doit être représentative de tous les systèmes présents et produire une remontée automatique des données vers GLPI.

Installation GLPI & plugin FusionInventory
GLPI = solution ITSM libre pour l'inventaire et les incidents

GLPI (Gestionnaire Libre de Parc Informatique) est déployé sur un serveur Linux. Le plugin FusionInventory (ou son successeur GLPI Agent) est installé sur chaque machine du parc pour la remontée automatique des inventaires.

Serveur Linux — Installation GLPI
# ── Prérequis LAMP ─────────────────────────────────────
root@srv-glpi:~# apt update && apt install -y apache2 mariadb-server php php-{mysql,curl,gd,mbstring,xml,zip,intl,ldap,apcu,xmlrpc}

# ── Créer la base de données GLPI ──────────────────────
root@srv-glpi:~# mysql -u root -p
MariaDB> CREATE DATABASE glpi CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
MariaDB> CREATE USER 'glpi'@'localhost' IDENTIFIED BY 'GlpiP@ss!';
MariaDB> GRANT ALL PRIVILEGES ON glpi.* TO 'glpi'@'localhost';

# ── Télécharger et déployer GLPI ───────────────────────
root@srv-glpi:~# wget https://github.com/glpi-project/glpi/releases/download/10.0.15/glpi-10.0.15.tgz
root@srv-glpi:~# tar xzf glpi-10.0.15.tgz -C /var/www/html/
root@srv-glpi:~# chown -R www-data:www-data /var/www/html/glpi
# Continuer l'installation via l'interface web : http://SRV-GLPI/glpi
Déploiement agent GLPI — client Linux
# ── Installation de l'agent GLPI sur client Linux ──────
user@client-linux:~$ curl -s https://raw.githubusercontent.com/glpi-project/glpi-agent/master/contrib/unix/glpi-agent-install.sh | sudo bash

# ── Configurer l'agent pour pointer vers le serveur GLPI
root@client-linux:~# nano /etc/glpi-agent/agent.cfg
# server = http://10.2.0.10/glpi   ← adresse du serveur GLPI

# ── Démarrer et activer l'agent ────────────────────────
root@client-linux:~# systemctl enable --now glpi-agent
root@client-linux:~# glpi-agent --force  # forcer une remontée immédiate

Mission 3 — Déploiement de l'agent GLPI
Mission 3Agent via GPO
Tous les systèmes TiersLieux86 doivent posséder l'agent GLPI. Pour les machines Windows, le déploiement est automatisé via une GPO (Group Policy Object) liée à l'AD. Les machines Linux sont configurées manuellement ou via un script.
GPOActive DirectoryMSIGLPI AgentPowerShell
1
Déploiement via GPO — Machines Windows
Installation automatique du MSI GLPI Agent sur tous les postes Windows du domaine

Une GPO de déploiement logiciel est créée dans l'AD pour installer automatiquement le package MSI de l'agent GLPI sur toutes les machines Windows membres du domaine, sans intervention manuelle.

1
Préparer le package MSI
Télécharger GLPI-Agent-x.x-x64.msi depuis GitHub → placer le fichier dans un partage réseau accessible par tous les postes (ex: \\DC\NETLOGON\GLPI-Agent.msi)
2
Créer la GPO de déploiement
Gestionnaire de stratégie de groupe → Créer une GPO → Configuration ordinateur → Paramètres du logiciel → Installation de logiciels → Nouveau → Indiquer le chemin UNC du MSI
3
Lier la GPO à l'OU appropriée
Lier la GPO à l'OU "Ordinateurs" (ou toute OU contenant les postes clients) → L'agent s'installera automatiquement au prochain démarrage des machines
4
Configurer le serveur GLPI
Via les options MSI ou un fichier de transform (.mst) : définir l'URL du serveur GLPI et l'intervalle de remontée (ex: toutes les 24h)

Mission 4 — Sécurité & sauvegarde des données
Mission 4HTTPS & Backup
Sécurisation des échanges GLPI via HTTPS (certificat SSL/TLS). Mise en place d'une sauvegarde automatisée complète et journalière des données GLPI et de l'inventaire pour garantir la continuité de service.
HTTPSSSL/TLSCronmysqldumpSauvegarde automatisée
1
Sécurisation HTTPS — Certificat SSL/TLS
Confidentialité des échanges entre GLPI et les utilisateurs

GLPI doit communiquer en HTTPS pour protéger les identifiants et données sensibles. Un certificat auto-signé est généré pour l'environnement de test ; en production un certificat Let's Encrypt serait utilisé.

Serveur GLPI Linux — Activation HTTPS
# ── Générer un certificat auto-signé (lab) ─────────────
root@srv-glpi:~# openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
    -keyout /etc/ssl/private/glpi.key \
    -out /etc/ssl/certs/glpi.crt \
    -subj "/CN=glpi.chasseneuil.tierslieux86.fr"

# ── Activer SSL dans Apache ─────────────────────────────
root@srv-glpi:~# a2enmod ssl
root@srv-glpi:~# a2ensite default-ssl

# ── Configurer le VirtualHost HTTPS (/etc/apache2/sites-available/glpi-ssl.conf)
<VirtualHost *:443>
    ServerName     glpi.chasseneuil.tierslieux86.fr
    DocumentRoot   /var/www/html/glpi
    SSLEngine      on
    SSLCertificateFile    /etc/ssl/certs/glpi.crt
    SSLCertificateKeyFile /etc/ssl/private/glpi.key
</VirtualHost>

# ── Rediriger HTTP → HTTPS ──────────────────────────────
root@srv-glpi:~# a2enmod rewrite
# Dans le VirtualHost *:80 : Redirect "/" "https://glpi.chasseneuil.tierslieux86.fr/"
root@srv-glpi:~# systemctl restart apache2

Sauvegarde automatisée journalière
Base de données GLPI + fichiers → cron quotidien

Une solution de sauvegarde complète et journalière est mise en place. Elle couvre la base de données MariaDB (via mysqldump) et les fichiers GLPI. Les sauvegardes sont conservées 7 jours avec rotation automatique.

Script de sauvegarde — /usr/local/bin/backup-glpi.sh
#!/bin/bash
# ── Script de sauvegarde GLPI (journalier) ─────────────
DATE=$(date +%Y%m%d_%H%M)
BACKUP_DIR="/var/backups/glpi"
DB_NAME="glpi"
DB_USER="glpi"
DB_PASS="GlpiP@ss!"
GLPI_DIR="/var/www/html/glpi"

# Créer le répertoire de sauvegarde du jour
mkdir -p $BACKUP_DIR/$DATE

# Sauvegarde de la base de données
mysqldump -u $DB_USER -p$DB_PASS $DB_NAME > $BACKUP_DIR/$DATE/glpi_db.sql

# Sauvegarde des fichiers GLPI
tar czf $BACKUP_DIR/$DATE/glpi_files.tar.gz $GLPI_DIR/files/

# Suppression des sauvegardes de plus de 7 jours (rotation)
find $BACKUP_DIR -type d -mtime +7 -exec rm -rf {} +

echo "Sauvegarde $DATE terminée avec succès" >> /var/log/backup-glpi.log
Planification via cron (chaque nuit à 2h00)
# ── Ajouter la tâche cron ───────────────────────────────
root@srv-glpi:~# chmod +x /usr/local/bin/backup-glpi.sh
root@srv-glpi:~# crontab -e
# Ajouter la ligne suivante :
0 2 * * * /usr/local/bin/backup-glpi.sh

# ── Vérifier la liste des crons actifs ─────────────────
root@srv-glpi:~# crontab -l

# ── Tester la restauration depuis une sauvegarde ───────
root@srv-glpi:~# mysql -u glpi -p glpi < /var/backups/glpi/20260421_020000/glpi_db.sql

Mission 5 — Haute-disponibilité HaProxy
Mission 5HaProxy & GLPI HA
Un second serveur GLPI est déployé. HaProxy distribue la charge entre les deux serveurs GLPI. En cas de panne de l'un, l'autre prend le relais automatiquement. La base de données est externalisée sur un 3e serveur partagé.
HaProxyLoad Balancer2 serveurs GLPIBDD externaliséeHealth checks
1
Architecture haute-disponibilité
HaProxy en frontal + 2 serveurs GLPI + 1 serveur BDD partagé

L'architecture HA permet de garantir la continuité du service GLPI même en cas de panne d'un serveur applicatif. La base de données est sur un serveur dédié vers lequel les deux serveurs GLPI pointent.

HaProxy — /etc/haproxy/haproxy.cfg
# ── Installation HaProxy ────────────────────────────────
root@haproxy:~# apt install -y haproxy

# ── Configuration /etc/haproxy/haproxy.cfg ─────────────
global
    log /dev/log local0
    maxconn 2000

defaults
    mode    http
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend glpi_front
    bind *:443 ssl crt /etc/haproxy/certs/glpi.pem
    default_backend glpi_servers

backend glpi_servers
    balance   roundrobin
    option    httpchk GET /glpi/index.php
    server glpi1 10.2.0.21:80 check inter 2s rise 2 fall 3
    server glpi2 10.2.0.22:80 check inter 2s rise 2 fall 3 backup

listen stats
    bind *:8404
    stats enable
    stats uri /haproxy-stats
La directive check inter 2s rise 2 fall 3 signifie : vérifier toutes les 2s, considérer le serveur UP après 2 succès consécutifs, DOWN après 3 échecs. Le mot-clé backup fait de glpi2 un serveur de secours activé uniquement si glpi1 est DOWN.
HaProxy→GLPI-SRV-1 (health check /glpi/index.php)À tester
HaProxy→GLPI-SRV-2 (backup activé si SRV-1 DOWN)À tester
Client Web→GLPI accessible via HaProxy après arrêt de SRV-1À tester
